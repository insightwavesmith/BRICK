## 3. 문제가 생겼을 때 (핵심 규율 — 땜빵 절대 금지)

1. **3축 문제정의가 항상 먼저.** 수리를 말하기 전에 어느 축의 문제인지부터 확정한다:
   - **Brick(계약)**: 계약의 틈인가, 명시됐는데 무시됐나?
   - **Agent(수행)**: 어느 레인이 최초 이탈했고 뒤 렌즈가 잡았나 방치했나?
   - **Link(이동/게이트)**: 거짓을 잡을 기계가 그 자리에 있었나?
   - **엔진(support)**: 콘텐츠 문제가 아니라 메커니즘 결함인가? (격리 재현 먼저)
   축 확정 전에 수리 발주 금지 — 가까운 support 명사를 패치하지 않는다.
2. **원인분석 → 해결, 순서 강제.**
   - 원인이 장부(증거 행·diff·재현)에서 **즉시 확정되면** → 원인 크기에 맞는 최소 수리 발주.
   - **원인이 바로 파악 안 되면 → 수리가 아니라 원인파악(조사) 빌딩을 굴린다.**
     COO가 추측으로 때우지 않는다. 조사 빌딩이 기전을 규명한 뒤에만 수리 순번.
     (실증 선례: resume 원장 불일치 — 조사 빌딩 2회로 트리거→기전 규명 후 수리 대기열 진입)
3. **단순한 해결만**: 원인 크기 = 수리 크기. 밴드에이드·증상 패치·모듈 증식·기능추가 반사 금지,
   단순화/삭제 우선(헌법 진단 5단). 이미 있는 메커니즘부터 찾는다.
4. **처방은 원인 축의 자기 선언 표면에 꽂는다**: Agent 원인=프롬프트/agent object,
   Brick 원인=반환 계약(return.yaml)/템플릿, Link 원인=게이트, 엔진 원인=결함 수리 빌딩.
5. **같은 계열 실패 3라운드 = 강제 정지.** 재발주 전에 반드시: ①문제가 몇 개의 독립 결정으로
   분해되는지 전수 열거 ②검증 렌즈가 각 항목을 개별 대조하는지 ③반복 자체가 엔진 결함
   증상인지. (llm= 8라운드 실사례 — QA concern 한 줄만 고쳐 재발주하는 패턴 금지)
6. 홀드는 처분 전 §3.0 4갈래 질문. 처분 클래스 확인(budget_exhaustion만 raise, 그 외 forward).
   같은 그래프 모양에서 resume 반복 실패 = 더 안 파고 fresh 재발주.
