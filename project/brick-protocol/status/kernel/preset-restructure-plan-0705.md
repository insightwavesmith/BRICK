출처: Smith 질문(0705 오후) "프리셋을 매우 잘 쓰게 구성하면 어떤가, 오히려 사고(思考)를
막을까?" → 조사자 판정: 위험의 본체는 프리셋이 아니라 **진입이 메뉴 선택이냐 질문이냐**
(0703 단일-레인 고착 사고는 프리셋 없이도 났다 · 헌법 Rule 4가 이미 이 실패 모드를 법으로
경계). 워크플로 레퍼런스는 완성 그래프 프리셋 대신 **품질 패턴(숙어)+자유 조립**을 쓴다 —
그 구조를 BRICK 저작 표면에 이식한다. **작업x — 발주-준비 문서, 시공은 형제 COO.**
graph-sizing-doctrine-refresh-0705.md의 R5로 연결된다. source truth 아님.

## 0. 원칙 한 줄

**프리셋은 사고(思考)의 어휘가 되게 하고, 메뉴가 되지 않게 한다.** 실측 근거:
selection_hint는 29/29 완비이나 anti-hint 0건, 전부 완성-그래프 단위, 진입 순서 미규정
(0705 측정) — 이 상태로 "잘 쓰게"만 만들면 메뉴 고착 쪽으로 기운다.

## 1. 3층 구조

### 1층 — 부품(블록) 카탈로그 신설 【신규】

재사용 단위를 완성 그래프에서 **스테이지 블록**으로 한 층 내린다. 블록 = 복붙 가능한
DSL 조각 + 언제/언제-아님 문서. **엔진·CLI 변경 없음** — materializer는 지금처럼 DSL만
받는다(Rule 3 2차엔진 금지 · Rule 9 compose_building 정본 · Rule 10 DSL 유지). 위치 제안:
`brick/templates/blocks/*.md` (프리셋과 분리된 저작 코퍼스).

| 블록 | DSL 뼈대 | 언제 | 언제 아님(anti) |
|---|---|---|---|
| B1 존-파티션 fan | `fan([read-only 존리더×K]) → 수렴` | 대형 읽기(레인당 ~1k줄), 존 경계를 계약에 박음 | 같은 파일에 쓰는 가지들(직렬로) |
| B2 검증 렌즈 fan | `fan(code-attack-qa · axis-attack-qa · evidence-integrity) → closure` | 저신뢰/계약·정책·보안 접촉 | 고신뢰 read-only(렌즈 0~1) |
| B3 심사 패널 | `fan(design×N 관점) → review 채점 → closure 채택` | 설계 선택지가 넓고 반증 불가한 판단 | 반증 가능한 사실 추출(1두뇌+반증 1렌즈) |
| B4 완결성 비평 꼬리 | closure 직전 `review("빠진 것 전수 — 모달리티·미검증 주장")` | 대형 조사/전수 감사 | 소형 단건 |
| B5 무인 척추 | `design→work→closure, gates=()` | 고신뢰 소형 | 계약/권위/보안 접촉 |
| B6 사람게이트 마감 | `gates=("human-review",)` + HOLD 후 공식 approve-from-hold 재개(gap2 v2) | 헌장·계약·고객 표면 변경 | 무인으로 닫아도 되는 read-only |
| B7 미지수 확장 | 출생 `expansion_node_budgets` 선언 → 걷다가 HOLD+승인 T10 확장 | 끝 모양이 불확실한 일 | 모양이 확정된 일(예산 선언은 소음) |
| B8 마를-때까지 라운드 | 계약 종료선에 "연속 2라운드 신규 발견 0 = 종료" 명시, COO가 라운드 발주 | 미지수 크기 탐색(버그 사냥·전수 스윕) | 엔진 루프화 금지 — 라운드 주체는 COO(무정지 팬 비목표) |

완성 프리셋 29개는 폐기하지 않는다 — **"블록 조립 예시"로 강등**하고 각 프리셋 머리에
구성 블록을 표기한다(예: recon-fleet = B1+B2+B4).

### 2층 — 질문-우선 진입 규율 【기존 R1 핀에 1줄 병합】

발주 진입은 항상: **①분할 가능한가 ②파티션 파일-비충돌인가 ③4축 다이얼 → 그 답을
selection_hint와 대조 → 맞으면 프리셋 채택, 아니면 블록 조립.** 헌장 사이징 핀(R1)에
한 줄 추가: "프리셋은 다이얼 답이 나온 뒤 고른다 — 메뉴에서 시작하지 마라." 같은 줄을
brick-task-author PRESET 입력 모드 절에도 각인(양 사본).

### 3층 — 프리셋 위생 【29개 전수】

1. **anti_hint 1줄 의무화**: selection_hint 아래 "Do NOT select when …" — 각 프리셋의
   실측/개연 오선택 사례로 작성(예: recon-fleet → "파티션 가능한 단일 대형 파일 조사면
   존-fan(B1) 조립이 우선"). 신규 프리셋 등록 시 anti_hint 없으면 반입 금지(체커는
   A+ W2 manifest와 합류 — 여기서 별도 체커 신설 안 함).
2. **블록 구성 표기**: 프리셋 머리에 `blocks: B?+B?` 한 줄.
3. **도그푸드 신호**: 빌딩 증거의 `build_input_mode`(preset_task vs graph)로 채택률을
   가끔 측정(조사자 수동 — 스케줄러 금지, Rule 3). 판독: **~100% 프리셋 = 사고 정지
   신호 / ~0% = 라이브러리 부패 신호 / 건강 = 중간.**

## 2. 하지 않는 것 (경계)

- 블록 실행기·블록 CLI·프리셋 추천 엔진 **신설 금지** — 블록은 문서+DSL 스니펫 코퍼스다.
- 프리셋 대청소(29개 통폐합)는 이번 범위 아님 — anti_hint를 달아보면 중복이 드러난다.
  통폐합은 그 측정 후 별도 판단.
- 블록에도 고착은 생긴다 — anti 열이 블록 표에도 있는 이유. Rule 4(축 보존 최소 그래프)는
  블록 위에서도 상위법이다.

## 3. 발주 모양 제안 (형제용 — 문서만, 1차 웨이브 코드와 비충돌)

```
슬라이스 S1: work(블록 카탈로그 8건 신설 + 프리셋 29건 anti_hint·blocks 표기,
             write_scope: brick/templates/blocks/** · brick/templates/presets/**)
             → fan( review(이 기획 §1 표와 대조 — 블록/anti 전수 실존),
                    inspect(경계 §2 — 엔진/CLI 표면 신설 0 확인) )
             → closure
슬라이스 S2: 진입 규율 1줄은 별도 발주 없음 — graph-sizing R1(헌장 핀)·R3(task-author)
             발주에 합류(중복 발주 금지).
```
