## 한 줄 결론

같은 습관이 네 곳에서 반복됐다: **"필드는 선언돼있고 배선도 됐는데, 어긴다고 실패하는 게이트(bite)가 없다"** 또는 **"이어질 수 있는 통로가 있는데 안 이어진다"**. Smith의 3층 규율(법/주입/강제)에서 항상 **③강제만 늦게 온다** — 그리고 세션이어짐은 그 강제 이전에 "애초에 기억을 잇는 통로 자체"가 빠진 케이스다.

## 개선 항목 4가지 (전부 동등하게 중요 — 우선순위는 발주 순서일 뿐, 중요도 순위 아님)

| 항목 | 영향/근거 | 한줄 원인 | 처방 규모 |
|---|---|---|---|

| **Brick** | 12개 빌딩 | write_scope를 산문으로만 "건들지마" 적어놓고 기계가 안 막음 | 중간 |

### 1. Link축 — 반려-재시도 배관 (Part1+2+3)

20건의 "SILENT"는 한 가지 병이 아니라 4가지 다른 이유(반려자체가없음/verification_gap이라 설계상 재시도불가/사람HOLD 방치/빌딩통째반려)로 각각 죽은 것. 예산·주소문법 규칙은 이미 대부분 고쳐졌고, 남은 건:
- **Part1**: 반려사유 문법검사를 배달시점→접수시점으로 당김
- **Part2**: 인용 하나 잘못 써도 엔진이 이미 아는 확실한 주소(concern_doc_ref)는 버리지 말고 배달
- **Part3**: 종이1(주소)만 배달하지 말고 같은 return 안 종이2(이미 압축된 요약)도 같이 실어서 배달. 종이1의 좁고 딱딱한 스키마는 유지(합치면 AI가 "관찰→성공"으로 물타기할 위험) — **배달할 때 둘 다 챙기는 것만** 고침
- **전달 세부규칙**: (a) Link의 라우팅 판단은 종이1만 읽음, 종이2는 다음 attempt 에이전트가 맥락 파악용으로 읽으라고 같은 봉투에 실릴 뿐 (b) 엔진은 종이2를 골라 담지 않고 전부 그대로 나름, 유일한 예외는 비밀정보 스크럽

### 2. Brick축 — write_scope 산문펜스 (Fix A/B/C)

`forbidden_paths` 필드도 있고 매처도 있는데, 정상편집 하나만 섞이면 금지경로 위반이 무제한으로 같이 통과됨(체커 자신이 "후속과제"라고 인정한 부분). 발주 스킬(SKILL.md)이 "넓게 선언+산문으로 자제"를 공식 권장해서 이 패턴을 계속 찍어내고 있었음.
- **Fix A**: 금지경로 위반방향 게이트 신설(기존 fake-landing 패턴 복제)
- **Fix B**: 발주 스킬 규칙 뒤집기 — 산문 불가침은 forbidden_paths에도 반드시 미러
- **Fix C**: 사람이 승인하는 reroute엔 `re_instruction` 필수화(COO 자동승인 경로는 그대로 둠)

### 3. Agent축 — 증명 파이프라인 (범위: work뿐 아니라 모든 브릭 종류)

WRITE(파일쓰기)는 선언→실측→비교→게이트 4단 파이프라인이 이미 있는데, PROOF(증명커맨드)는 그 짝이 아예 없어서 에이전트 자기보고를 아무도 검증 안 함. 그래서 압박받으면 인접한 쉬운 커맨드로 몰래 대체해도 다 통과.
- **범위 정정(대화로 확정)**: fable5 원문이 "an agent"라고 썼듯, 이 구멍은 work뿐 아니라 **QA를 포함한 모든 브릭 종류**의 자기보고에 다 해당됨
- **게이트 결과 정정(대화로 확정)**: 원래 fable5 설계는 HOLD(사람대기)였는데, 이러면 Link축 1위 문제(HOLD가 밤새 방치됨)를 그대로 재현함 — **HOLD 아니라 reroute(implementation_gap, 정식 반려)로 만들어서 오늘 고친 Link 파이프라인(Part1+2+3)을 그대로 타게 함.** HOLD는 재시도 예산 다 쓰고도 안 되면 최후수단으로만.
- **누가 재시도 대상이 되나**: 자기보고와 실측이 어긋난 그 브릭 자신(work 코드가 틀리면 work로, QA의 자기보고가 틀리면 QA로) — 이건 에이전트가 자기 자신을 지목하는 게 아니라 **게이트(Link)가 실측된 사실만 보고 독립적으로 결정**하는 것이라 "자기지목 금지" 규칙과 무관함

### 4. 세션이어짐 — 같은 빌딩 내 attempt 연속성

지금 attempt N→N+1은 완전히 새 세션(새 대화)으로 뜬다. 워크트리(코드)는 이미 `building_id` 단위로 이어지는데, 세션(기억)만 안 이어진다.
- **실측**: `session_continuity_mode`가 4개 값(`none`/`continue_if_available`/`start_or_continue`/`fork_from_available`)으로 선언은 돼있는데, 실제 배선(`adapter_local_cli.py`)은 `none`일 때 플래그 하나 붙이는 것뿐 — 나머지 3개는 배선 자체가 없음
- **CLI는 이미 지원함**: `codex resume`, `claude -c/--continue/--resume/--fork-session` 전부 확인됨 — BRICK이 안 쓰고 있을 뿐
- **선결 조건**: codex는 공유 `~/.codex` SQLite 단일-writer 락 때문에 세션영속화를 기본 꺼둠(동시빌딩 데드락 방지) — 켜려면 이 동시성 문제부터 해결
- **다른 항목과의 관계**: Link Part1~3(주소+요약 배달)과 경쟁이 아니라 보완 — 세션이 이어지면 프롬프트에 다시 박을 필요가 줄어들지만, 세션이 안 이어지는 동안(또는 안 되는 케이스)엔 Link의 배달 처방이 안전망 역할

## 구체 사례 — Agent축 증명 파이프라인이 실제로 도는 모양

```
브릭(선언) → 링크(발사) → 에이전트(실행) → 링크게이트(실측·대조)
                                                │
                    ┌───────────────────────────┴───────────────────────────┐
                    ▼ 반려: 주소+요약 실어 재시도                            ▼ 충분: 진행
              브릭(같은 노드, 재처리)                                    브릭(기록)
                    │                                                       │
                    └──(링크 발사 → 에이전트 재실행) ──┐                     ▼
                                                       │                 링크(다음으로) → ...
                                                       └─────────────────┘
```
세 개의 "링크" 상자가 색이 같은 이유: 서로 다른 기능이 아니라, **같은 링크 역할이 매 이동마다 다시 등장**하는 것(Brick=노드, Link=엣지, 모든 이동은 엣지를 지남 — 예외 없음).

## 증류에서 빠진 것 (참고, 우선순위 낮음)
- write_scope 너무 좁음(5개 빌딩, returns-persistence 계열) — 반대방향 버전
- 형제 빌딩 write_scope 충돌(2개 빌딩) — 표본 부족

## 발주 순서 (발주하기 쉬운 순서일 뿐, 중요도 순서 아님)

```
1차: Link Part1+2+3 묶음 (같은 파일군, 가장 작고 국소적)

3차: Agent 증명 파이프라인 (범위: 모든 브릭 종류, 결과: reroute)
별도: 세션이어짐 (동시성 문제 선결 필요 — 별도 빌딩 추천, 중요도는 위 3개와 동급)
```
