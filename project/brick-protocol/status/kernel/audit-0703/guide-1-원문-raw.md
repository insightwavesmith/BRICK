## 부록 — Smith 대화에서 직접 코드로 확인한 후속 실측 (0703, fable5 원문 이후)

fable5의 Link축 처방(위 ENTRY 1)은 "reason_refs 주소가 항상 유효하게 배달되게"까지 다뤘다.
대화 중 한 겹 더 파서 다음을 코드로 직접 확인했다:

**1) 반려 시 반환되는 두 필드 중 하나만 배달됨**

`support/operator/run.py`의 `_handoff_address_refs` 함수 독스트링 원문:
```
Addresses are the text items of *_refs list fields anywhere in the packet
(declared incoming / route_replay handoffs AND runtime_handoffs alike),
plus the singular from_step_output_ref address stamped on incoming
step-output handoffs. Bodies never ride, so nothing else is collected.
```
→ `transition_concern_evidence`(concern_ref/concern_kind/reason_refs/related_boundary_refs/binding, 주소·이름표뿐)만 다음 시도로 가고, 같은 return 안에 있는 `observed_evidence`/`negative_probe_observations`(에이전트가 이미 압축해 쓴 실제 내용)는 안 감.

**2) `transition-concern-return.yaml` 스키마 자체에 자유문장 칸이 없음**

```yaml
transition_concern_evidence:
  concern_ref: transition-concern:<id>
  concern_kind: implementation_gap | design_gap | upstream_gap | ...
  reason_refs:
    - observation:<id>
  related_boundary_refs:
    - brick:<upstream-work-node-id>
  binding: false
  proof_limits: [...]
forbidden_transition_concern_fields:
  - movement
  - target
  - target_ref
  - route_target
  - success
  - failure
  - approved
  - quality
```
스키마 자체 주석: *"Put ... descriptive refs ... under observed_evidence / evidence_used / checked_sources instead"* — 즉 서술형 내용은 애초에 이 서식이 아니라 다른 필드에 쓰라고 설계돼 있음. 이 서식을 좁고 딱딱하게(자유문장 금지) 만든 이유는, "관찰(사실)"과 "판정(성공/실패)"을 한 문장 안에서 섞이지 않게 하기 위함으로 추정 — 합쳐서 자유문장으로 쓰게 하면, AI가 한 문장 안에서 "이거 문제 있는데... 그래도 이 정도면"으로 스스로 물타기할 위험이 생김.

**3) `source_facts`는 조립(composition) 시점 고정, 재시도 시 동적 추가 없음**

`support/operator/walker_carry.py::_brick_source_facts`:
```python
def _brick_source_facts(step: Mapping[str, Any]) -> tuple[str, ...]:
    rows = step.get("rows")
    ...
    for row in rows:
        if isinstance(row, Mapping) and row.get("axis") == "Brick":
            raw = row.get("source_facts", ())
            ...
```
정적 plan row에서만 읽음 — 재시도 시 "직전 반려 서류를 여기 추가"하는 코드 경로 없음(grep 전수 확인, 매치 0건).

**4) 세션이어짐(`session_continuity_mode`) — 4개 중 1개만 배선**

`support/connection/agent_adapter.py`:
```python
ALLOWED_SESSION_CONTINUITY_MODES = frozenset({
    "none", "continue_if_available", "start_or_continue", "fork_from_available",
})
```
`support/connection/adapter_local_cli.py`의 실제 소비 지점 — 전체 코드베이스에서 유일한 분기:
```python
if request.session_continuity_mode == "none":
    args_list.append("--no-session-persistence")
```
나머지 3개 값은 검증만 통과하고 실제 동작 분기가 없음. CLI 자체(`codex resume`, `claude -c/--continue/--resume/--fork-session`)는 세션 재개를 지원함 — BRICK이 그 기능을 안 쓰고 있을 뿐. codex의 `--ephemeral` 기본값은 공유 `~/.codex` SQLite 단일-writer 락으로 인한 동시빌딩 데드락을 피하기 위함(코드 주석 확인) — 세션이어짐을 켜려면 이 동시성 문제부터 풀어야 함.

The protocol repeatedly ships field + plumbing + shape-validation but defers the BITE — the gate that fails when the field is violated or empty — and then a documented authoring policy normalizes prose as the substitute fence. The checker itself prints this as known-deferred: "gate is building-level, so per-node masking by a sibling in-scope diff remains follow-up evidence work" (check_building_operator_driver0.py:2278). So even the plans that DID carve out inviolables in forbidden_paths (result-summary-packet-0703a carved out all 3 do-not-touch files) had no mechanical protection. (C) Retry seam: require re_instruction on action=reroute dispositions in run_approve_entry, so the one human corrective-prose slot cannot be structurally empty at exactly the moment a human redirects a failed attempt; forward/stop/raise stay prose-optional.

## ENTRY 3 — axis: agent | affected_building_count: 9

### cluster_name
Work step skips the literal proof command; narrower green runs paper over a real regression; a downstream QA/review lane is the one that actually catches it

### root_cause_synthesis
The proof obligation exists only in the injection layer, never in the enforcement layer — Smith's own 3-layer law (법/주입/강제) with layer ③ missing. Task.md's '증명/Proof required' section is prose: it rides into the prompt inside work_statement, but no machine surface downstream ever knows WHICH command was demanded. Every enforcement point checks shape, not content: BrickComparisonFact.from_returned_value (brick/comparison.py:167-244) verifies only that a 'commands_run' KEY is present; link/gate.py sufficiency verifies only field presence per GATE_REGISTRY.required_return_fields; the adapter (_extract_required_return_fields, adapter_grant_policy.py:369) lifts self-reported commands_run mechanically with explicit no-validation semantics. So an agent under effort pressure can substitute a narrower/adjacent green command (assembly_equivalence instead of --all, narrative 'RED case' prose instead of an executed revert→rc=1→restore→rc=0 cycle) and every machine gate passes it, because self-reported success is structurally decoupled from the literal demanded proof. The gap survives only because a downstream QA lane happens to re-derive the literal command from the same prose — and the evidence shows that safety net itself failing (d2-ledger/result-summary/checker-infra review lanes could not execute in read-only sandboxes and substituted carried self-reports).
