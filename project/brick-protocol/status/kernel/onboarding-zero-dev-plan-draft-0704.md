# 온보딩 기획 초안 — "개발 1도 모르는 고객" 기준 재설계 (0704, Smith 지시)

저자 = 조사자 세션(기획 초안). 온보딩 세션의 3축 기획(onboarding-plan-draft, 축1 시공
중)과의 관계 = **충돌 아님, 상위층**: 3축은 현 깔때기의 마찰을 없애는 기반 공사(전부
유효, 선행), 이 문서는 그 깔때기에 **입장조차 못 하는 다수 고객**을 위한 층을 새로 긋는다.
증거 한계: 기획 초안. source truth·발주 계약 아님 — 발주 설계는 Smith 방향 확정 후 별도.

## 한 줄 결론

**"문서를 더 쉽게"로는 이 고객에 못 닿는다 — 읽고 따라하는 주체를 고객에서 에이전트로
바꾸는 것이 BRICK다운 답이다.** 고객 몫은 3가지로만 줄인다: ①계정 만들기 ②구독/결제
③"예/아니오" 대답. 나머지 전부(설치·진단·복구)는 에이전트가 한다.

## 1. 페르소나 (누구 기준으로 짓나)

| 페르소나 | 실체 | 비율 가정(Smith) | 현 깔때기 진입 |
|---|---|---|---|
| **P0 무개발** | 터미널을 열어본 적 없음. "클론"이 뭔지 모름. 자사몰 사장님/마케터 | **대부분** | ❌ 불가능 |
| P1 반개발 | 터미널 복붙은 함. 에러 나면 멈춤 | 일부 | 🚧 마찰 다수(=3축이 고치는 층) |
| P2 개발자 | 현 README로 충분 | 소수 | ✅ |

## 2. 현 깔때기의 숨은 전제 지식 — 전수 인벤토리 (실물 근거)

README "시작하기 (2분)"와 install.sh가 **말하지 않고 요구하는 것들**. P0에게는 하나하나가
탈락 지점이다:

| # | 숨은 전제 | 실물 근거 | P0에게는 |
|---|---|---|---|
| 1 | GitHub 계정 + **포크 개념** | README "BRICK은 포크해서 쓰는 구조" | "포크가 뭐죠" |
| 2 | gh CLI 설치 + `gh auth login` 완료 | install.sh 3단계 preflight | gh가 뭔지 모름 |
| 3 | `{OWNER}` 치환 (중괄호 포함 제거) | README 경고문이 실존할 정도로 실패 다발 | 치환 개념 없음 |
| 4 | 터미널 열기·복붙·엔터 | 전 과정 | 첫 관문부터 공포 |
| 5 | 경로/환경변수(BRICK_HOME, PYTHONPATH) | install.sh/README | 이해 불가 |
| 6 | python3/uv/pipx 생태계(실패 시 진단이 이 어휘) | install.sh 1·2·5단계 | 외계어 |
| 7 | provider CLI 설치 + 구독 + API 키 | onboard 위자드(codex/claude/gemini) | 카드 결제와 키의 관계 모름 |
| 8 | 실패 메시지 독해(ModuleNotFoundError 등) | README failure signal 절 | 즉시 포기 |

관찰: 3축 기획은 #2 일부(#gh 재시도 안내)·#3(자리표시자 제거)·#6(pipx 충돌)·#8(친절
메시지)을 다듬는다 — 옳고 필요하다. 그러나 #1·#4·#7(계정·터미널·구독)은 문서 개선으로
해소가 **원리상 불가능**한 층이다.

## 3. 계층 전략 — L0~L3

```
L3  제품화(원클릭 앱/호스팅)          ← Smith 결정 영역, 이 초안 범위 밖 (비용 큼)
L2  에이전트 동반 설치  ★이 초안의 제안 ← 고객은 채팅만, 설치는 에이전트가
L1  복붙 한 줄 완전자동                ← 터미널 1회는 감수하는 P1 하한 끌어올림
L0  개발자 깔때기(현재)                ← 3축 기획이 다듬는 중 (선행, 전부 유효)
```

- **L1 (복붙 한 줄 완전자동)**: 지금은 "gh 설치돼 있고 로그인돼 있으면" 한 줄이다.
  L1의 정의 = **전제조건까지 스크립트가 해결**: gh 부재 시 설치 유도, 미로그인 시
  `gh auth login` 브라우저 흐름 안내, {OWNER}는 로그인된 계정에서 자동 추론(치환 개념
  자체 제거), 실패 시 한국어 한 줄 처방(이미 install.sh의 설계 언어 — 확장). 설치는
  안 막힘 유지(축3 결정 계승), 안내만 추가.
- **L2 (에이전트 동반 설치) — 이 초안의 핵심 제안**: P0은 문서를 읽지 않는다. 대신
  이미 쓰는 채팅 AI(Claude 등)에게 **"이 주소를 열고 시키는 대로 해줘"** 한 문장만
  전달하게 한다. 씨앗은 이미 repo에 있다 — README의 "AI/운영자가 그대로 확인할 줄"
  블록(command/expected/failure signal 3행 형식)이 정확히 에이전트-판독용 표면이다.
  L2 = 이 표면을 **정식 계약으로 승격**: ①AGENT-ONBOARD.md 단일 진입 문서(에이전트가
  읽는 설치 명세: 단계·기대출력·실패신호·복구처방 전부 기계 판독형) ②고객에게 묻는
  질문은 3종만 통과(계정 유무 / 구독 의사 / 예·아니오 확인) ③doctor·진단 출력도 같은
  형식으로(에이전트가 읽고 고객에겐 결론만 통역). BRICK 철학의 자기 적용: **온보딩
  자체가 "에이전트가 하는 일"이다.**
- **L3 (제품화)**: 데스크톱 앱, 호스팅, 스토어 배포 등. 투자 규모가 다른 층 — Smith가
  사업 판단으로 열 때만. 이 초안은 자리만 표시.

## 4. 설계 원칙 3개 (모든 층 공통)

1. **읽는 주체 교체**: 사람용 문서(README)와 에이전트용 명세(AGENT-ONBOARD)를 분리.
   P0 대상 표면에서 전문용어는 "설명"이 아니라 **제거**가 목표(에이전트 쪽으로 이동).
2. **고객 몫 3가지 원칙**: 계정 / 결제 / 예·아니오. 이 셋 밖의 행동을 고객에게
   요구하는 단계가 남아 있으면 그 단계가 다음 개선 대상이다 — 온보딩 완성도의 측정자.
3. **막히면 에이전트가 진단**: 실패 메시지는 고객용(한 줄 결론)과 에이전트용(진단
   데이터) 이중 표면 — 축2의 message_ko 방향을 계승·확장.

  다음 조사 후보(지금은 조사자·형제 세션의 개발자 재현뿐 — P0 실측 0건이 이 초안의
  최대 증거 한계다).

## 6. 실행 순서 제안 (발주 설계 아님 — 방향만)

1. **(진행 중) 3축 = L0/L1 기반 공사** — 그대로 완주 (축1 시공 중, 축3 결정 반영).
2. **L1 스크립트 승격** — gh 부재/미로그인/OWNER 자동 추론. 축1 완료 직후가 자연 순서.
3. **L2 에이전트 명세** — AGENT-ONBOARD.md + doctor 이중 표면. README의 AI 확인 줄
   블록을 씨앗으로 확장. **체커 동반**(T9 원칙): 명세의 command/expected가 실물과
   어긋나면 RED(온보딩 문서가 썩는 것을 기계가 감시).
4. **P0 실측 3~5명** — L2까지 깔린 뒤 실제 무개발 고객으로 TTFB·이탈 측정 → 결과로
   L3 여부를 Smith가 판단.

## 7. Smith 결정 대기 (이 초안이 못 정하는 것)

1. L2까지 간다 / L1에서 멈춘다 / L3까지 연다 — 층 선택.
2. P0 기본 provider 추천(비용 구조가 고객마다 다름 — 구독 vs API 키).
3. 대상 언어: 한국어 전용인가(현재 혼재), P0 표면 한국어 통일 여부.
4. Windows 우선순위(축1에 "큼"으로 등재됨 — P0 중 Windows 비율이 판단 재료).
