# 온보딩 기획 초안 v2 — "개발 1도 모르는 고객" 기준 재설계 (0704, Smith 지시)

v1은 fable5 적대 비평에서 FLAWED(치명 2·중 9·소 5) — 전면 개정. v1의 치명 결함:
①L2 순환(터미널 실행 가능한 AI를 갖추는 일 자체가 관문 #7) ②L1의 {OWNER} 제거 불가
(치환은 스크립트 입수 **전**의 clone 줄에 있음 — 호스팅 설치 URL 부재 + 비공개 repo).
③선행 확정 설계(install-ux-design-0618.md "깔때기 역전 — 첫 초록불 먼저, 결제는 업셀")
미계승. v2는 셋 다 정면 반영. 온보딩 세션 3축 기획과의 관계 = 기반 공사(선행)이나,
축2(사람용 메시지 개선)와는 긴장 있음 — §6에 정직 기술. 기획 초안이며 발주 계약 아님.

## 한 줄 결론 (v2)

**P0(무개발 고객)의 관문을 "소비자급 앱 설치 1회 + 구독 1건"으로 압축하고, 그 이후의
모든 개발자 행위(저장소 받기·설치·진단·복구)를 그 앱 안의 에이전트가 대신한다.**
결제는 가치 확인 뒤(깔때기 역전 계승) — 무료 첫 초록불이 먼저다.

## 1. 페르소나 (v1 유지 + 하중 가정 명시)

P0 무개발(터미널 무경험, 대부분 — **가정, 실측 0건**) / P1 반개발(복붙 가능) / P2 개발자.
**하중 가정 2개는 실측 1순위**: ①P0이 정말 대부분인가 ②P0의 OS 분포(Windows 비중) —
이 둘이 아래 층 설계의 기초 하중이라, **실측 전에 비싼 층을 짓지 않는다**(§5 순서).

## 3. 계층 전략 v2

```
L3  제품화(원클릭/호스팅)        ← Smith 사업 판단, 자리만
L2  에이전트 동반(재정의) ★     ← 관문을 "앱 1 + 구독 1"로 압축, 이후 전부 에이전트
L1  스크립트 강화(축소 재정의)   ← 배포 결정 없이는 상승분 제한적 — 정직 축소
L0  개발자 깔때기(현재)          ← 3축 기획이 다듬는 중(선행)
```

**L2 재정의 — v1 순환의 해소**: "고객이 이미 쓰는 웹챗"은 터미널을 못 만진다(v1 오류).
대신 **에이전트 런타임 확보 자체를 소비자급 1관문으로 고정**한다: Claude 데스크톱 앱
(맥/윈도 설치형 — 더블클릭 설치, npm 불요)을 깔고 구독 1건 → 그 안의 에이전트가
이후 전부(gh 준비·저장소 받기·install.sh·진단)를 동반 수행. 순환이 "개발자 관문 7종"
에서 "일반 앱 설치+구독"으로 줄어드는 것이 이 층의 전부이며, **이 1관문을 P0이 정말
통과하는지가 실측 1번 질문이다**(통과 못 하면 L2 기각, L3 또는 운영자 동반으로).
- 에이전트 명세(AGENT-ONBOARD): 단계·기대출력·실패신호·복구·**재개 상태**(중단 후
  다음 세션이 어디까지 됐는지 아는 표면) 기계 판독형. **선행 필수 = T9 체커**(명세의
  command/expected가 실물과 어긋나면 RED) — 비평 렌즈 실측: README의 AI 확인 줄이
  이미 썩어 있음(expected "5) 설치 점검 완료" vs 실물 "5) brick 진입점 설치 완료 ✅",
  install.sh:186). **씨앗은 있으나 살아있지 않다 — 체커 없인 이 층 전체가 같은 방식으로
  썩는다.**
- **보안 경계(신설, v1 공백)**: ①진입은 "아무 URL"이 아니라 사관학교가 주는 고정
  문구+고정 주소 1개(인젝션 습관 훈련 방지) ②에이전트 금지 행위 계약 명문화 — 결제
  실행 대행 금지, API 키 생성 대행 금지, 명세 밖 curl|sh 신뢰 판단 금지, 인증(브라우저
  로그인)은 항상 사람 몫 ③명세는 서명/체커로 무결성 감시.
- **gh 로그인의 정직한 분류**: 브라우저 8자리 코드 입력은 에이전트가 대신 못 한다 —
  "고객 몫"을 3종에서 4종으로 정정: **계정 / 로그인 승인(브라우저) / 결제 / 예·아니오**.

**L1 축소 재정의 — v1 자기모순의 해소**: 현 install.sh는 이미 gh 부재·미로그인·pipx
부재를 한국어로 안내한다(:78-84, :139-149) — "안내 추가"는 상승분 0에 가깝다(v1 오류).
{OWNER} 자동 추론은 **배포 결정(§7-1) 이전엔 원리상 불가**(치환이 clone 줄에 있음).
따라서 L1의 정직한 범위 = 3축 완주 + **에이전트 없이도 죽지 않는 실패 처방의 완결성**
(모든 실패 출구에 "다음 한 줄" 존재 — 축2와 공동)까지만. 그 이상은 배포 결정 후.

## 4. 깔때기 역전 계승 (v1 누락 — 선행 확정 설계)

install-ux-design-0618.md 확정: **첫 초록불을 먼저 보여주고, 그다음 "진짜로 만들래?"를
묻는다.** 무료 경로(provider 없이 local/verify green)가 이미 실존(README:61-63).
v2 지표 2단 분리:
- **TTFG**(Time-To-First-Green): 첫 접촉 → 무료 첫 초록불. **결제 관문 없음.** P0 목표
  30분(에이전트 동반 기준).
- **결제 전환**: 초록불 이후 provider 구독 전환율 — 별도 지표, 별도 관문(이탈 측정 대상).
v1의 "구독 포함 30분 TTFB"는 폐기(가치 확인 전 결제 강요 — 선행 설계 위반이었음).

## 5. 실측 우선 — 순서 재설계 (v1 역순 정정)

v1은 L2를 다 짓고 실측(가장 비싼 층을 가장 검증 안 된 가정 위에) — 뒤집는다:
1. **(지금) 3축 완주** — L0/L1 기반. + 비평 렌즈 실측 발견 1건 즉시 수리 이관:
   README AI 확인 줄 expected 불일치(위 §3) — 축2/3 범위.
2. **가이드 실측 3~5명(싼 검증 먼저)**: L2를 짓기 **전에**, 운영자(사람) 동반으로 실제
   P0 수강생이 현 깔때기+시안(종이 프로토타입 수준의 에이전트 대본)을 통과하는지 관찰.
   측정 방법 명시: 동석 관찰+화면 녹화(텔레메트리 없음 — n=3~5는 '통과율'이 아니라
   **'이탈 지점 관찰'**로 표현 정정). 하중 가정 2개(P0 비율·OS 분포)와 "앱 1+구독 1"
   관문 통과 가능성을 여기서 판정.
3. **실측 결과로 분기**: 통과 → L2 시공(AGENT-ONBOARD+체커) / 앱 관문도 못 넘음 →
   L2 기각, 운영자 동반 온보딩(사람 서비스)을 정식 층으로 승격 + L3 판단 재료로.
4. **설치 이후 생애주기(신설)**: 갱신은 install.sh 재실행의 git pull뿐(:129-131) —
   P0에게 갱신 신호를 누가 주나, ff 실패는 누가 푸나 → L2 명세의 정기 점검 절 +
   지원 채널(아래)로 배정.
5. **지원 채널(신설)**: 에이전트가 진단 불능일 때 P0이 두드릴 사람 경로(카톡/채널 —
   사관학교 운영 자산 재사용)와 응답 기대치 명시. 실측 중 막힌 고객 수용처이기도 함.

## 6. 3축과의 관계 — 정직 기술 (v1 "충돌 없음" 정정)

축1·축3: 그대로 선행, 충돌 없음. **축2는 긴장 있음**: 축2는 사람이 읽는 단일 메시지
개선인데, L2가 서면 이중 표면(고객=결론 한 줄 / 에이전트=진단 데이터)으로 재편된다 —
축2 산출물의 독자는 P1로 좁아지고 일부는 재작업 대상. 축2 발주 시 이 방향을 알고
쓰도록 온보딩 세션에 통보(중복 투자 방지).

## 7. Smith 결정 대기 (v2 — 배포 결정이 승격됨)

1. **배포 표면(신규, 최우선)**: 비공개 repo 정책 유지 전제에서 — (a)AGENT-ONBOARD
   명세·설치 진입점을 공개 호스팅할 최소 표면을 여나(무엇까지 공개?) (b)고객 접근권은
   누가 만드나(사관학교가 초대/포크 대행? 별도 고객 배포 채널?). **이 결정 없이는
   L1 자동추론도 L2 도달 경로도 원리상 못 연다** — v1이 이 결정을 목록에 안 올린 것이
   치명 결함이었다.
2. 층 선택: 실측(§5-2) 후 L2 시공 여부 — 지금 정할 건 "실측 먼저" 승인뿐.
3. P0 기본 provider 추천(구독 vs API 키 — 결제 전환 관문 설계와 결부).
4. Windows: P0 다수 OS일 개연성 — **부록이 아니라 하중벽**. 실측(§5-2)에서 OS 분포
   확인 후, macOS 선행/Windows 병행을 결정.
5. P0 표면 한국어 통일(외부 영어 표면 ⑭는 통제 밖임을 전제로).

증거 한계: 기획 초안 v2. P0 실측 0건이 여전한 최대 한계이며, v2의 구조 자체가 그
한계를 §5 순서(싼 검증 먼저)로 흡수하도록 짜였다. source truth·발주 계약 아님.
